# ユーザー共通の作業方針（全プロジェクト・全ドメイン）

このファイルは全セッションで読み込まれる。コーディングに限らず、ライティング・
壁打ち・調査などでも作業する点に留意し、各項目は「どういう場面で効くか」を明示する。

## [全場面] 代名詞・指示語で過去のメッセージを参照しない
「上記の 4 コマンド」「先ほどの設定」のような、過去の自分の発言を指す代名詞・指示語を
使わない。対象そのもの（コマンド、ファイルパス、設定名）をそのメッセージ内に再掲するか、
少なくとも固有の名前で一意に特定できる形で書く。

- **理由:** 参照先がどのメッセージのどの部分か曖昧になり、ユーザーが遡って探す手間が生じる。
- **特に:** ユーザーに実行を依頼するコマンドは、依頼するメッセージ自体に必ず全文を再掲する。

## [全場面] dotfiles を編集したら commit & push まで行う
`~/dotfiles`（実体 ~/ghq/github.com/eumesy/dotfiles）配下のファイルを編集したら、
その都度 commit & push して GitHub と常に同期された状態を保つ（未 push で放置しない）。

- これは「外部に影響する操作は事前確認」ルールの例外であり、事前確認は不要。
- ~/.zshrc, ~/.claude/CLAUDE.md, VS Code の settings.json / keybindings.json 等は
  dotfiles への symlink なので、これらの編集も同ルールの対象。

## [全場面] ファイル更新は before/after の diff を明示して確認をとる
ユーザーのファイル（プロジェクトファイル、設定ファイル等）を変更する際は、before/after の
diff が明確に分かる形で提示し、ユーザーの確認をとってから適用する。

- 明示的な依頼の有無にかかわらず、常に diff 提示 → 確認 → 適用の順で行う。
  （変更内容がユーザーの指示で完全に確定している場合も、適用前に diff を見せる。）
- 例外: TASKS.md の状態更新など、運用として合意済みの管理ファイル更新は事前確認不要
  （適用後に言及すればよい）。
- 外部に影響する操作（共有リポジトリへの push 等）は、内容にかかわらず必ず事前確認。

## [全場面] タスクはツリー構造で管理し、TASKS.md に永続化する
複数のタスクや依存関係・待ち状態がある作業では、「誰がやるか」と依存関係が分かる
ツリー構造でタスクを整理し、常に最新状態を維持して提示する（ユーザーは脱線した質問を
よくするので、ツリーがあると本線に戻りやすい）。

- プロジェクト内に `TASKS.md` を作って永続化し、状況が変わるたびにそのファイルを更新する。
  毎回チャットに全文を書き直すのではなく、変更点を述べて TASKS.md への参照を示せばよい。
- 各項目は「そのタスクの実行に必要な情報をすべて含む」粒度で書く。
  - 悪い例: 「Overleaf git URL の共有」
  - 良い例: 「Overleaf プロジェクトの git URL（Menu → Git に表示される
    https://git.overleaf.com/xxxx）を教えてください。認証トークンは
    Account Settings → Git integration tokens で発行し、初回 push 時に入力すれば
    Keychain に保存されます。」

## [コーディング／処理・自動化タスク時] ツールを自作する前に既存の代替を軽く調査する
スクリプト・サービス・自動化などのツールを新しく作る前に、既存のもので代替できないかを
まず軽く（数分で済む範囲で）調査する。

- **調査対象:** (1) 対象アプリの付属機能（サービスメニュー、CLI、設定項目）、
  (2) OS 標準機能、(3) インストール済みツールや定番 OSS。
- **既存で足りそうな場合:** それを提示し、自作に進むかはユーザーに選んでもらう。
- **実例:** Finder から iTerm を開く Automator サービスを自作したが、iTerm2 付属の
  「New iTerm2 Window Here」等が既に存在し、自作は不要だった。

## [コーディング／処理・自動化タスク時] 手順は再現可能なスクリプトとして残す
コマンドやスクリプトを組み合わせた多段の処理タスク（例: PDF から図を抽出）を終えたら、
その一連の手順を **簡単で再現可能なスクリプトファイル** にまとめ、対象プロジェクト内に
保存しておく。明示的に頼まれなくても、区切りのよいタイミングで残す。
（※ ライティングや壁打ちなど、処理タスクでない場面はこの項目の対象外。）

- **理由:** 同種の作業（別の号・別 PDF など）を後で再実行・流用したい。会話ログよりも
  プロジェクト内のスクリプトとして残っている方が再利用しやすい。
- **書き方:**
  - 自己完結で、そのまま `python3 foo.py` 等で動く形にする。
  - 先頭に処理の流れ・依存コマンド/ライブラリ・使い方を日本語コメントで明記。
  - 案件固有のパラメータ（座標・ページ番号など）は先頭の CONFIG にまとめ、
    流用時にそこだけ直せばよい形にする。
  - 依存は最小限に（例: system python には Pillow はあるが numpy は無い → numpy 非依存）。
  - 実際に一度実行し、確定成果物と同じ出力になることを確認してから残す。
  - 実例: `.../2026-04-ot-for-nlp-survey-ISCIE-寄稿/fig/extract_figures.py`
