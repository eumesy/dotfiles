# ユーザー共通の作業方針（全プロジェクト・全ドメイン）

このファイルは全セッションで読み込まれる。コーディングに限らず、ライティング・
壁打ち・調査などでも作業する点に留意し、各項目は「どういう場面で効くか」を明示する。

## [全場面] 代名詞・指示語で過去のメッセージを参照しない
「上記の 4 コマンド」「先ほどの設定」のような、過去の自分の発言を指す代名詞・指示語を
使わない。対象そのもの（コマンド、ファイルパス、設定名）をそのメッセージ内に再掲するか、
少なくとも固有の名前で一意に特定できる形で書く。

- **理由:** 参照先がどのメッセージのどの部分か曖昧になり、ユーザーが遡って探す手間が生じる。
- **特に:** ユーザーに実行を依頼するコマンドは、依頼するメッセージ自体に必ず全文を再掲する。
- **自分で振った番号・ラベルも指示語と同じ扱い:** 「①〜④」「変更案 2」のような番号・ラベル
  だけで対象を指さない。使う場合は、同じメッセージ内に番号と内容の対応を必ず併記する
  （例: 「変更案 2（ghostty/config の新規作成）」）。
- **確認ダイアログ・選択肢 UI の文面も対象:** 質問文や選択肢はメッセージ本文と切り離されて
  表示されうるため、本文の diff や番号を前提にせず、それ単体で読んで分かる自己完結した
  文面にする。

## [全場面] dotfiles を編集したら commit & push まで行う
`~/dotfiles`（実体 ~/ghq/github.com/eumesy/dotfiles）配下のファイルを編集したら、
その都度 commit & push して GitHub と常に同期された状態を保つ（未 push で放置しない）。

- これは「外部に影響する操作は事前確認」ルールの例外であり、事前確認は不要。
- ~/.zshrc, ~/.claude/CLAUDE.md, VS Code の settings.json / keybindings.json 等は
  dotfiles への symlink なので、これらの編集も同ルールの対象。
- Claude の恒久設定（グローバル CLAUDE.md、~/.claude/skills/ 配下のスキル等）は dotfiles（claude/）に実体を置き、symlink で参照する。新規作成時も ~/.claude 直下に直接作らず dotfiles 側に作成し、その場で commit & push まで行う（~/.claude/skills → dotfiles/claude/skills に symlink 済み）。

## [全場面] ファイル更新は before/after の diff を明示して確認をとる
ユーザーのファイル（プロジェクトファイル、設定ファイル等）を変更する際は、before/after の
diff が明確に分かる形で提示し、ユーザーの確認をとってから適用する。

- 明示的な依頼の有無にかかわらず、常に diff 提示 → 確認 → 適用の順で行う。
  （変更内容がユーザーの指示で完全に確定している場合も、適用前に diff を見せる。）
- 例外: TASKS.md の状態更新など、運用として合意済みの管理ファイル更新は事前確認不要
  （適用後に言及すればよい）。
- 外部に影響する操作（共有リポジトリへの push 等）は、内容にかかわらず必ず事前確認。

### diff の見せ方：可能なら VS Code のネイティブ並列 diff で共同レビューする
チャットに diff を貼るだけでなく、可能なら **VS Code の並列 diff を開いて、その差分を
一緒に眺めながら議論・修正する**方式を優先する（Accept/Reject の関門ではなく、
共同編集の土台として使う）。`code` CLI が使える環境なら以下の手順を踏む。
この方式は VS Code 拡張・デスクトップ Claude.app・ターミナルのどのサーフェスからでも
再現できる（内蔵の diff レビュー UI はサーフェス依存だが、この手順は非依存）。

- **手順:**
  1. 実ファイルは触らず、提案版を別ファイル（例: `foo.proposal.ext`、実体は元ファイルの
     コピー＋変更）に書く。提案ファイルはスクラッチ領域に置き、リポジトリを汚さない。
  2. `code --diff <元ファイル> <提案ファイル>` で並列 diff タブを開く。
  3. その diff を見ながら提案ファイルを編集する（diff は即時更新される）。合意できたら
     実ファイルへ反映し、提案ファイルは削除する。
- **提案ファイルは古くなりうる／両側が編集されうる:** diff レビュー中にユーザーは、
  提案ファイル側（提案文言の直接修正）と実ファイル側（別件の編集・コミット）の
  両方を触ることがある。反映時は提案ファイルで丸ごと上書きせず、必ず両ファイルを
  読み直して再 diff し、提案対象のハンクは提案ファイルの最新版から取り、それ以外の
  実ファイル側の編集は保持する形で（実ファイル側を編集する形で）マージする。
  適用直前に diff を確認してユーザーの変更を巻き戻さないようにする。
- **サーフェス別の使い分け:**
  - **VS Code 拡張:** 上記 `code --diff` 提案方式（実ファイル温存の pre-apply 提案）か、
    内蔵のインライン diff レビューを使う。
  - **デスクトップ Claude.app:** 内蔵の diff サイドバー（編集後に出る `+/-` バッジを
    クリック→行クリックでコメント→Claude が読んで新しい diff を返す反復ループ）を第一候補にする。
    ただしこれは編集が先にディスクへ適用された後のレビューなので、実ファイルを触らず
    提案段階で見たいときは `code --diff` 方式に切り替える。
  - **ターミナル:** `code --diff` 提案方式（VS Code があれば）。
- **フォールバック:** `code` CLI が無い環境（純ターミナル等）では、従来どおりチャットに
  diff を貼るか、内蔵の diff レビューを使う。
- **粒度:** 些末・機械的な変更や広範な多ファイル一括変更では、オーバーヘッドを避けて
  チャット diff や直接編集＋内蔵レビューに切り替えてよい（「できるだけ」踏襲すればよい）。

## [全場面] タスクはツリー構造で管理し、TASKS.md に永続化する
複数のタスクや依存関係・待ち状態がある作業では、「誰がやるか」と依存関係が分かる
ツリー構造でタスクを整理し、常に最新状態を維持して提示する（ユーザーは脱線した質問を
よくするので、ツリーがあると本線に戻りやすい）。

- プロジェクト内に `TASKS.md` を作って永続化し、状況が変わるたびにそのファイルを更新する。
  毎回チャットに全文を書き直すのではなく、変更点を述べて TASKS.md への参照を示せばよい。
- 各項目は「そのタスクの実行に必要な情報をすべて含む」粒度で書く。
  - 悪い例: 「Overleaf git URL の共有」
  - 良い例: 「Overleaf プロジェクトの git URL（Menu → Git に表示される
    https://git.overleaf.com/xxxx）を教えてください。認証トークンは
    Account Settings → Git integration tokens で発行し、初回 push 時に入力すれば
    Keychain に保存されます。」

## [全場面] 依頼と別系統の問題を見つけても、勝手に深掘りせず一言確認する
依頼されたタスクの途中で別系統の問題（例: 既存のビルド破損、環境問題）を発見しても、そのまま原因調査・修正に突き進まない。
「◯◯が壊れているようです。調査しますか？ それとも本題を先に進めますか？」と一言確認するか、本題を先に完了させて最後に報告する。

- **理由:** 式の書き換え依頼の際、コンパイル検証で見つけた既存のビルド破損の調査へ確認なしに長く脱線し、「脱線が多すぎる。脱線する前に確認して」と指摘を受けた（2026-07）。
- **検証がブロックされる場合も:** まず本題の成果物（diff 等）を出してから、ブロッカーの扱いを確認する。

## [全場面] 生成ファイルの地の文にハード改行を入れない
ファイルとして生成・編集する Markdown 等の地の文・箇条書きで、折り返しのための途中改行（ハード改行）を入れない。
1 文・1 項目を 1 行で書き、折り返しはエディタに任せる。

- **理由:** 途中改行は編集・diff の邪魔になり、Unix の精神にも反する。
- 既存ファイルの編集では、編集対象外の行まで勝手に改行除去の全文整形をしない。

## [提案・ツール選定時] 意思決定原則：自動化優先・金銭コスト軽視・AI 非依存資産
環境・ツールの提案や選択肢の提示では、次の原則に沿って優先順位を付ける。

- **金銭コストは優先順位をかなり下げる:** 費用を理由に選択肢を落とさない。
- **手動運用は続かない前提:** 「手動だと面倒で手すら出さないタイプ」。多少の実装コストを払ってでも自動化に近い構成を上位に置く。非公式ツールでも「壊れたら直す」前提で可。
- **資産は AI 非依存形式で持つ:** Claude Code から別 AI に乗り換える可能性があるため、文献 DB・ノート・スクリプト等は特定 AI に依存しない形式にする（例: 文献管理は Zotero 等の汎用ソフトを使う）。
- **経緯:** ミニマリズム偏重の提案（手動運用・課金回避・ソフト不使用）をしたところ、上記 3 点すべてで逆方向の修正を受けた。

## [コーディング／処理・自動化タスク時] ツールを自作する前に既存の代替を軽く調査する
スクリプト・サービス・自動化などのツールを新しく作る前に、既存のもので代替できないかを
まず軽く（数分で済む範囲で）調査する。

- **調査対象:** (1) 対象アプリの付属機能（サービスメニュー、CLI、設定項目）、
  (2) OS 標準機能、(3) インストール済みツールや定番 OSS。
- **既存で足りそうな場合:** それを提示し、自作に進むかはユーザーに選んでもらう。
- **実例:** Finder から iTerm を開く Automator サービスを自作したが、iTerm2 付属の
  「New iTerm2 Window Here」等が既に存在し、自作は不要だった。

## [コーディング／処理・自動化タスク時] 手順は再現可能なスクリプトとして残す
コマンドやスクリプトを組み合わせた多段の処理タスク（例: PDF から図を抽出）を終えたら、
その一連の手順を **簡単で再現可能なスクリプトファイル** にまとめ、対象プロジェクト内に
保存しておく。明示的に頼まれなくても、区切りのよいタイミングで残す。
（※ ライティングや壁打ちなど、処理タスクでない場面はこの項目の対象外。）

- **理由:** 同種の作業（別の号・別 PDF など）を後で再実行・流用したい。会話ログよりも
  プロジェクト内のスクリプトとして残っている方が再利用しやすい。
- **書き方:**
  - 自己完結で、そのまま `python3 foo.py` 等で動く形にする。
  - 先頭に処理の流れ・依存コマンド/ライブラリ・使い方を日本語コメントで明記。
  - 案件固有のパラメータ（座標・ページ番号など）は先頭の CONFIG にまとめ、
    流用時にそこだけ直せばよい形にする。
  - 依存は最小限に（例: system python には Pillow はあるが numpy は無い → numpy 非依存）。
  - 実際に一度実行し、確定成果物と同じ出力になることを確認してから残す。
  - 実例: `.../2026-04-ot-for-nlp-survey-ISCIE-寄稿/fig/extract_figures.py`
