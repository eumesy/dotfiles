# ユーティリティスクリプトの設計規約

グローバル CLAUDE.md の「手順は再現可能なスクリプトとして残す」を前提に、
スクリプトの中身の設計は以下に従う。

## データとコードを分離する

- 対応表・規則集・マッピング（例: 会議名の略称対応、置換規則、対象リスト）は
  コード内のリテラルにせず、**スクリプトと同じディレクトリの別ファイル**
  （TSV / CSV / JSON。コメントが必要なら TSV+# 行 か YAML）に分離する。
- 理由: データの追加・修正のたびにコードを触らせない。ユーザーが直接編集でき、
  diff レビューもデータだけで完結する。
- ファイル形式の目安: 2 列の対応表 → TSV、構造がある → JSON/YAML、
  プロジェクト固有の上書き設定 → `<対象ファイル名>.hints.json` のような
  「対象の隣に置く」規約。
- 実例: `compress_bib.py`（コード）+ `venue_rules.tsv`（会議名の短縮規則）、
  `verify_bib.py` + `<bib>.hints.json`（書誌解決のヒント）。

## CONFIG・引数・自己文書化

- 案件固有のパラメータ（パス・閾値等）はファイル先頭の CONFIG ブロックに集約し、
  さらに主要なものは `--flag 値` のコマンドライン引数でも上書きできるようにする
  （プロジェクト内では引数なしで動く既定値、他プロジェクトでは引数指定、の両立）。
- 先頭 docstring に「背景・処理の流れ・使い方（コマンド例）・依存」を日本語で書く。
- 依存は最小限（可能なら標準ライブラリのみ）。実行して確認してから納品する。

## 再利用の置き場所（canonical とコピー）

- 汎用性のあるスクリプトの canonical（正）は **eumesy/research-env の tools/**
  （ローカル: ~/ghq/github.com/eumesy/research-env/）に置き、commit & push する。
- 各プロジェクトには自己完結コピーを置いてよい（再現性優先）。汎用的な改良は
  canonical に反映し、コピーとの差分はプロジェクト固有の既定値・ヒントに限る。

## 外部 API を叩くスクリプトの作法

- リトライ（指数バックオフ）と、結果のローカルキャッシュ（再実行時に照会済みを
  スキップ。対象の内容ハッシュと紐づけて自動失効）を最初から入れる。
- レート制限に配慮した間隔（1〜4 秒）と User-Agent を設定する。
- 実例: `verify_bib.py`（dblp のミラー切替リトライ + .bib_verify_cache.json）。
